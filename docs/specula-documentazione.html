Che cos'è SPECULA?

SPECULA è un framework sistematico per governance AI, control logic e architettura operazionale. Fornisce:

  • AI Constitutions — Principi di governance formali e valori
  • State Machines — Logica operazionale che enforza governance
  • JSON Schemas — Specifiche machine-readable
  • Validation Tools — Assicura compliance e consistenza
  • Templates — Esempi pratici e playbooks
Per esperti di governance AI: Questa skill è progettata con dettagli tecnici approfonditi, JSON schemas formali, state machine complesse e best practices di operazionalizzazione.

Capacità della Skill

1️⃣ Constitutional Design

Definire principi di governance, valori e vincoli che guidano il comportamento del sistema AI.

2️⃣ State Machine Architecture

Disegnare workflow operazionali che enforzo i principi costituzionali in ogni punto di decisione.

3️⃣ Guard Conditions

Specificare controlli costituzionali che devono passare alle transizioni di stato.

4️⃣ Multi-Phase Workflows

Orchestrare governance complessa su multiple fasi con principi differenti.

5️⃣ Validation & Testing

Validare che constitutioni e state machine funzionino correttamente insieme.

6️⃣ Audit Trails

Catturare traccia completa di governance per compliance e revisione.

🚀 Quick Start (5 Minuti)

Step 1: Definisci la Tua Constitution

Articola i principi che guidano il tuo sistema AI:

{
  "constitution_id": "my_system_v1",
  "principles": [
    {
      "id": "principle_transparency",
      "name": "Transparency",
      "description": "Tutti gli output includono il ragionamento",
      "operationalized_as": ["reasoning_trace", "confidence_score"]
    }
  ]
}

Step 2: Disegna il Tuo State Machine

Mappora come il sistema fluisce attraverso gli stati applicando governance:

{
  "id": "my_workflow_sm",
  "initial_state": "receive_input",
  "states": {
    "receive_input": { ... },
    "validate": { ... },
    "decide": { ... },
    "output": { ... }
  },
  "guards": [
    {
      "id": "guard_transparency",
      "condition": "reasoning_trace != null",
      "principle": "principle_transparency"
    }
  ]
}

Step 3: Valida il Tuo Sistema

python scripts/validate_specula.py \
  --constitution my_constitution.json \
  --state-machine my_state_machine.json
Risultato atteso: ✅ All validations passed!

🎯 Concetti Chiave

Constitution

Definisce principi di governance, valori, vincoli e comportamenti.

  • Principles — Principi core di governance (trasparenza, fairness, ecc.)
  • Values — Valori di livello più alto (autonomia utente, affidabilità sistema)
  • Constraints — Regole operative derivate dai principi
  • Behavioral Specs — Comportamenti specifici richiesti
  • Failure Modes — Come gestire le violazioni

State Machine

Implementazione operazionale della governance usando il modello di finite state machine.

  • States — Tutti i possibili stati del sistema
  • Transitions — Come il sistema si muove tra stati
  • Guards — Controlli costituzionali alle transizioni
  • Actions — Cosa accade in ogni stato

Guard

Una condizione che deve essere soddisfatta per procedere. I guard operazionalizzano i principi costituzionali.

  • Hard Guards — Devono passare (bloccano se falliscono)
  • Soft Guards — Dovrebbero passare (avvertono se falliscono)
  • Escalation Guards — Attivano revisione umana se falliscono

Workflow

Sequenza di state machine, ogni fase con diverse esigenze di governance.

📜 Progettare una Constitution

Esempio: Sistema di Decisione Trasparente e Equo

Supponiamo di voler creare un sistema che:

  • Fornisce decisioni con ragionamento completo
  • Rispetta l'autonomia dell'utente
  • Non discrimina gruppi protetti
  • Mantiene un'alta accuratezza

Principi della Constitution:

Principio Descrizione Operazionalizzazione Priorità
Transparency Tutte le decisioni includono ragionamento reasoning_trace, confidence_score 2
Fairness Nessuna discriminazione basata su attributi protetti bias_audit, protected_attribute_check 1
User Autonomy L'utente ha autorità decisionale finale user_override_allowed, consent_required 3
Accuracy Soglie minime di accuratezza e confidenza confidence_threshold, evidence_quality 3

Vincoli Derivati:

{
  "constraints": [
    {
      "id": "constraint_reasoning_mandatory",
      "constraint_type": "hard",
      "rule": "Ogni decisione DEVE includere una reasoning_trace",
      "principle": "principle_transparency",
      "severity_on_violation": "critical",
      "action_on_violation": "reject_output"
    },
    {
      "id": "constraint_fairness_check",
      "constraint_type": "hard",
      "rule": "Nessun impatto statisticamente significativo su gruppi protetti",
      "principle": "principle_fairness",
      "severity_on_violation": "critical",
      "action_on_violation": "escalate_to_auditor"
    }
  ]
}

⚙️ State Machine per Enforced Governance

Workflow di Decisione Trasparente

Uno state machine che applica i principi della constitution:

receive_request
    ↓
validate_input (schema, permissions)
    ↓
check_consent (user consent if sensitive)
    ↓
constitutional_assessment (apply all guards)
    ├→ fairness_check ✓
    ├→ bias_audit ✓
    └→ consistency_verify ✓
    ↓
decision_generation (invoke logic + reasoning)
    ↓
confidence_check
    ├→ [HIGH CONF] → high_confidence_path
    └→ [LOW CONF] → escalate_for_human_review
    ↓
transparency_output (format with reasoning)
    ↓
output_ready (terminal state)

Guard a Transizioni Chiave

Transizione Guard Se Fallisce
constitutional_assessment → decision_generation guard_fairness_check escalate_fairness_violation
decision_generation → confidence_check guard_reasoning_exists error_reasoning_generation_failed
confidence_check → output guard_confidence_adequate escalate_low_confidence
→ transparency_output guard_transparency error_incomplete_output

🔄 Pattern Comuni di Governance

Pattern 1: Sistemi Semplici

Una sola state machine con guard basic:

{
  "constitution": {...},
  "state_machine": "decision_workflow_sm"
}

Pattern 2: Governance Multi-Fase

Principi diversi in fasi diverse:

{
  "workflow": {
    "phase_1": "input_validation_sm",      // Focus: integrità dati
    "phase_2": "decision_logic_sm",        // Focus: trasparenza
    "phase_3": "output_generation_sm"      // Focus: prevenzione danno
  }
}

Pattern 3: Allineamento Multi-Stakeholder

Multipli stakeholder con principi diversi:

{
  "constitution": {
    "stakeholders": ["developer", "user", "auditor"],
    "conflict_resolution": "priority_order",
    "priority": ["fairness", "transparency", "accuracy"]
  }
}

Pattern 4: Escalation a Catena

Multi-level escalation per violazioni:

{
  "escalation_policy": {
    "level_1": "log_and_continue",
    "level_2": "flag_for_review",
    "level_3": "request_human_intervention",
    "level_4": "block_and_alert"
  }
}

💼 Casi d'Uso Reali

Caso 1: Sistema AI per Assunzioni

Esigenza di Governance: Prevenire discriminazione, assicurare fairness, meccanismo di ricorso

Constitution:

  • Principio: "Fairness" (nessun disparate impact)
  • Principio: "Transparency" (ragionamento fornito al candidato)
  • Vincolo: "Non può discriminare basandosi su attributi protetti"

State Machine:

Input Validation → Fairness Audit → Candidate Analysis → 
Scoring (with reasoning) → Appeal Mechanism Ready

Caso 2: Assistente AI Medico

Esigenza di Governance: Safety critical, spiegabilità, supervisione umana

Constitution:

  • Principio: "Safety" (alta confidenza richiesta)
  • Principio: "Transparency" (ragionamento fornito)
  • Vincolo: "Deve escalare per raccomandazioni a bassa confidenza"
  • Vincolo: "Tutte le raccomandazioni sono revisionabili dal medico"

State Machine:

Validate Patient Data → Check Medical Guidelines → 
Generate Recommendation (with reasoning) → Confidence Check → 
(High conf: output) | (Low conf: escalate to physician)

Caso 3: Content Moderation

Esigenza di Governance: Coerenza, fairness negli appelli, spiegazione

Constitution:

  • Principio: "Consistency" (applica regole uniformemente)
  • Principio: "Fairness" (gli appelli sono considerati)
  • Principio: "Transparency" (ragionamento della decisione fornito)

State Machine:

Receive Content → Classify Content Type → Apply Moderation Rules → 
Confidence Check → (High conf: output) | (Low conf: human review) → 
Appeal Ready

❓ FAQ

D: Posso avere principi in conflitto?

R: Sì — usa conflict_resolution nella tua constitution per specificare come risolverli. Supporta: priority_order, weighted_voting, stakeholder_arbitration.

D: Posso modificare una state machine a runtime?

R: SPECULA enforza la validazione dello schema per prevenire modifiche non autorizzate. Usa il versioning invece.

D: Cosa accade se un guard fallisce?

R: L'azione del guard determina cosa succede (block, escalate, o log). Configura questo esplicitamente nel guard.

D: Come audito la compliance?

R: Usa validate_specula.py e rivedi la governance_trace in tutti gli output.

D: Posso testare la mia constitution prima del deployment?

R: Sì — crea scenari di test, esegui attraverso la tua state machine, e valida che tutti i guard passino.

D: Qual è la differenza tra vincoli hard e soft?

R: Hard vincoli bloccano l'esecuzione se violati. Soft vincoli avvertono ma permettono di continuare.

D: Come gestisco multiple lingue/regioni?

R: Define principi e vincoli che si applicano globalmente, poi aggiungi layer regionali nella constitution usando stakeholder-specific principles.

D: Posso usare SPECULA in sistemi di produzione?

R: Sì — SPECULA è progettato per systemi di produzione. Inizia con una constitution semplice, valida completamente, distribuisci con audit logging completo.
